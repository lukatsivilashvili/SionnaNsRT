# Sionna ns-3 to Linux TAP proof-of-chain

This example sends UDP traffic generated by an ns-3 application from an LTE UE across a Sionna RT-managed LTE hop, through the EPC PGW, over a CSMA link to a ghost node, and then to Linux through TapBridge.

The current CNAM defaults are:

```text
eNB / TX / car_2: 49, -17, 12
UE  / RX / car_1 sequence:
  32, -9, 1.5
  -72, -31, 1.5
  -120, 90, 1.5
  -208, -101, 1.5
  205, 177, 1.5
```

The example moves the UE through the coordinate list every 20 seconds by default.

## 1. Clean Up Old Server

If a previous Sionna server is still holding UDP port `8103`, stop it first:

```bash
lsof -nP -t -i:8103 | xargs -r kill
```

If it refuses to exit:

```bash
lsof -nP -t -i:8103 | xargs -r kill -9
```

## 2. Start Sionna RT

Run this from the repository root:

```bash
cd src/sionna
python3 sionna_v1_server_script.py \
  --local-machine \
  --frequency=2.1e9 \
  --path-to-xml-scenario=scenarios/cnam_scene_new/cnam_scene.xml \
  --export-radio-map \
  --radio-map-dir=../../sionna-radio-maps \
  --radio-map-summary-csv=../../sionna-radio-maps/radio_map_summary.csv \
  --radio-map-tx-car-id=2 \
  --radio-map-rx-car-id=1 \
  --radio-map-camera-mode=rx-orbit \
  --radio-map-camera-rx-distance=130 \
  --radio-map-camera-distance-scale=1.05 \
  --radio-map-camera-side-offset=-60 \
  --radio-map-camera-height=140 \
  --radio-map-camera-height-scale=0.55 \
  --radio-map-camera-rx-target-weight=0.7 \
  --radio-map-camera-fov=78 \
  --radio-map-camera-look-at-height=20 \
  --radio-map-tx-power-dbm=23 \
  --radio-map-noise-figure-db=7
```

The script name in this tree is `sionna_v1_server_script.py`.

## 3. Build

In another terminal, from the repository root:

```bash
./ns3 build
```

## 4. Run ns-3 With TAP

Recommended: let ns-3 create `sionna-tap0`, then pause the simulation briefly so you can start `tcpdump` manually. This avoids the timing problem where the TAP interface does not exist until ns-3 has already started.

Do not run `sudo ./ns3`. This tree refuses to run the wrapper as root. Use `--enable-sudo` so the wrapper can request privileges for TapBridge.

```bash
./ns3 run --enable-sudo "sionna-ns3-to-linux-tap \
  --tapMode=ConfigureLocal \
  --tapName=sionna-tap0 \
  --simTime=105 \
  --appStartTime=3 \
  --manualCapturePauseSeconds=20 \
  --enableSystemTapPcap=false \
  --uePositionInterval=20 \
  --enablePcap=true \
  --pcapPrefix=sionna-ns3-to-linux-tap-csma \
  --enablePacketSummary=true \
  --packetSummaryFile=sionna-ns3-to-linux-tap-packet-summary.csv \
  --enableStaticNeighborCache=true \
  --enbPosition=49,-17,12 \
  --uePositions=32,-9,1.5;-72,-31,1.5;-120,90,1.5;-208,-101,1.5;205,177,1.5"
```

`simTime=105` gives the fifth coordinate at `t=80s` enough time to generate traffic and trigger the final heatmap.

`enableStaticNeighborCache=true` pre-populates ARP on the simulated PGW-to-TAP CSMA link. This avoids losing the first ns-3-generated UDP packets while the PGW waits for the Linux TAP side to answer ARP.

`manualCapturePauseSeconds=20` pauses the run after TapBridge creates the Linux TAP and before UDP traffic starts. Increase it if you want more time to start `tcpdump`.

By default the example sends `SHUTDOWN_SIONNA` when ns-3 finishes, so the Sionna server should exit after the run. To keep it alive across multiple ns-3 runs, add:

```bash
--shutdownSionna=false
```

## 5. Capture Linux TAP Traffic

When ns-3 prints the manual capture pause message, run this in another terminal:

```bash
ip addr show sionna-tap0
sudo tcpdump -i sionna-tap0 -n -vv udp port 9000 -w sionna-tap0-udp9000.pcap
```

Stop `tcpdump` with `Ctrl+C` after ns-3 finishes. You can also inspect live traffic without saving:

```bash
sudo tcpdump -i sionna-tap0 -n -vv udp port 9000
```

Open the saved capture later:

```bash
tcpdump -nn -r sionna-tap0-udp9000.pcap | head
wireshark sionna-tap0-udp9000.pcap
```

In `ConfigureLocal` mode, `sionna-tap0` is created by ns-3, so `tcpdump` cannot start before ns-3. The manual pause solves this by stopping the simulation after TapBridge has created the interface and before UDP traffic begins.

Optional UDP listener:

```bash
nc -ul 9000
```

## 6. Expected Outputs

ns-3 writes CSMA/TAP-side PCAP files by default:

```text
sionna-ns3-to-linux-tap-csma-*.pcap
```

These PCAPs are generated by ns-3 on the simulated PGW-to-ghost CSMA link. They are useful for checking the packets immediately before/at the TapBridge egress path. Use `--enablePcap=false` to disable them or `--pcapPrefix=<name>` to change the filename prefix.

The Linux-side capture from `tcpdump -i sionna-tap0 ... -w sionna-tap0-udp9000.pcap` is separate. That file captures what appears on the real Linux TAP interface.

The example starts and stops that Linux-side capture automatically by default:

```text
sionna-tap0-udp9000.pcap
```

Automatic Linux-side capture is available, but the manual pause workflow above is currently the recommended path on this machine. Automatic capture is controlled by `--enableSystemTapPcap`, `--systemTapPcapStartTime`, `--systemTapPcapFile`, and `--systemTapPcapUseSudo`. The example uses non-interactive `sudo -n tcpdump`, so it will not wait for a password prompt. The automatic capture intentionally does not use a `udp port 9000` capture filter; it captures all TAP frames so ARP/control traffic and unexpected packet types are visible during debugging. Filter for UDP port 9000 during analysis instead.

ns-3 also writes a per-coordinate packet summary by default:

```text
sionna-ns3-to-linux-tap-packet-summary.csv
```

This CSV has one row per UE coordinate/time window. It includes the UE coordinate, the fixed eNB coordinate, TX/RX distance, UDP application packets/bytes transmitted, packets/bytes observed on the ghost/TAP-side CSMA device, delivery ratio, first/last packet times, and mean TAP-side inter-arrival time. Join it with `sionna-radio-maps/radio_map_summary.csv` by the `index` column to compare packet behavior with Sionna path gain/loss, SNR, RTT, LOS, and heatmap artifacts.

Radio-map outputs are written under:

```text
sionna-radio-maps/
```

For each unique UE/eNB coordinate pair, expect:

```text
*_scene.png          Native Sionna scene-rendered heatmap
*_grid.png           2D top-down fallback heatmap
*_stats.png          Per-coordinate analysis plot
*_path_gain_db.npy   Raw path-gain grid in dB
*.json               Metadata for that coordinate
```

The run-level CSV is:

```text
sionna-radio-maps/radio_map_summary.csv
```

The CSV includes TX/RX coordinates, TX/RX distance, RX nearest-cell path gain, RX path loss, estimated RX power, noise floor, SNR, geometric delay/RTT, ray-traced delay/RTT when available, LOS status when available, map min/mean/max path gain and path loss, camera position, and file paths for the generated artifacts.

Each `*_stats.png` contains:

```text
Simple path-gain profile sampled along the direct TX-to-RX line
Compact key-metrics summary with distance, path gain/loss, RX power, SNR, RTT, and LOS
```

The distance axis in `*_stats.png` is not a movement trace. It is the distance along the direct line from the eNB/TX to the UE/RX for that one exported coordinate. The UE is not gradually moved from the previous coordinate to the next one; ns-3 schedules discrete `SetPosition` updates at `t=0s`, `20s`, `40s`, and so on. Each heatmap is therefore one static coordinate pair, and the graph's maximum distance changes because each UE coordinate is a different distance from the fixed eNB.

```bash
./ns3 run "sionna-ns3-to-linux-tap \
  --enableTap=false \
  --sionna=false \
  --simTime=45 \
```



```bash

  --sionna=false \
  --enableTap=false \
  --simTime=15 \
  --appStartTime=3 \
  --enableUePositionSequence=false"
```

Try automatic TAP capture instead of the manual pause:

```bash
./ns3 run --enable-sudo "sionna-ns3-to-linux-tap \
  --tapMode=ConfigureLocal \
```



## Manual TAP Mode

If `ConfigureLocal` cannot create the TAP in your environment, use `UseLocal`:

```bash
sudo ip tuntap add dev sionna-tap0 mode tap user $USER
sudo ip addr add 10.10.2.2/24 dev sionna-tap0
sudo ip link set sionna-tap0 up
./ns3 run --enable-sudo "sionna-ns3-to-linux-tap \
  --tapMode=UseLocal \
  --tapName=sionna-tap0 \
  --simTime=105"
```

## ns-3 Example Arguments

Common ns-3 arguments:

```text
--enableTap=true|false
    Enable TapBridge egress to Linux.
--tapMode=ConfigureLocal|UseLocal
    ConfigureLocal lets ns-3 create sionna-tap0. UseLocal uses a pre-created TAP.
--tapName=sionna-tap0
    Linux TAP interface name.
--sionna=true|false
    Enable/disable Sionna RT propagation.
--localMachine=true|false
--sionnaServerIp=127.0.0.1
    Sionna server location.
--shutdownSionna=true|false
    Ask Sionna server to exit when ns-3 finishes.
--simTime=105
    Total simulation duration.
--appStartTime=3
    UDP application start time. Keep this after the manual capture pause point.
--packetSize=512
--interval=100ms
--trafficApp=UdpClient|OnOff
--dataRate=40960bps
--port=9000
--maxPackets=0
    UDP traffic controls. maxPackets=0 derives the count from simTime and interval.
--enbPosition=49,-17,12
--uePositions=32,-9,1.5;-72,-31,1.5
--enableUePositionSequence=true|false
--uePositionInterval=20
    LTE node coordinates. uePositions is a semicolon-separated list of x,y,z triples.
--enablePcap=true|false
--pcapPrefix=sionna-ns3-to-linux-tap-csma
    ns-3-side CSMA/TAP-facing pcap output.
--enablePacketSummary=true|false
--packetSummaryFile=sionna-ns3-to-linux-tap-packet-summary.csv
    Per-coordinate packet CSV.
--enableSystemTapPcap=true|false
--systemTapPcapStartTime=1
--systemTapPcapFile=sionna-tap0-udp9000.pcap
--systemTapPcapUseSudo=true|false
    Optional automatic Linux tcpdump. Manual pause is usually easier.
--enableStaticNeighborCache=true|false
    Pre-populate ARP entries on the simulated PGW-to-TAP CSMA link.
--manualCapturePauseSeconds=0
    Wall-clock pause after TAP creation so manual tcpdump can be started.
--enableFlowMonitor=true|false
--flowMonitorFile=sionna-ns3-to-linux-tap.flowmon.xml
    Optional FlowMonitor XML.
--txPower=23
--earfcn=100
    LTE radio controls.
```

For `uePositions`, pass a semicolon-separated list of `x,y,z` triples. Make `simTime` long enough for the last coordinate:

```text
simTime >= 1 + uePositionInterval * number_of_ue_positions
```

## Sionna Heatmap Arguments

Common Sionna server heatmap arguments:

```text
--export-radio-map
--radio-map-dir=../../sionna-radio-maps
--radio-map-summary-csv=../../sionna-radio-maps/radio_map_summary.csv
--radio-map-tx-car-id=2
--radio-map-rx-car-id=1
--radio-map-cell-size 5 5
--radio-map-size 500 500
--radio-map-center 0 0 1.5
--radio-map-samples=50000
```

Native scene render camera arguments:

```text
--radio-map-camera-mode=rx-orbit
--radio-map-camera-rx-distance=130
--radio-map-camera-distance-scale=1.05
--radio-map-camera-side-offset=-60
--radio-map-camera-height=140
--radio-map-camera-height-scale=0.55
--radio-map-camera-rx-target-weight=0.7
--radio-map-camera-look-at-height=20
--radio-map-camera-fov=78
--radio-map-show-orientations
```

With `rx-orbit`, the camera frames a point biased toward the UE/RX, backs away along the eNB-to-UE direction, and scales its distance and height with TX/RX separation. This zooms out automatically for far-away UE coordinates while keeping the receiver closer to the center of the native scene render. Set `--radio-map-camera-rx-target-weight=0.5` for exact midpoint framing, or closer to `1.0` to center the RX more strongly.

For identical framing in every image:

```text
--radio-map-camera-mode=fixed
--radio-map-camera-position 220 -360 220
--radio-map-camera-look-at 0 0 0
```

Other render options:

```text
--radio-map-resolution 756 570
--radio-map-render-samples=128
--radio-map-vmin=-140
--radio-map-vmax=-40
--radio-map-tx-power-dbm=23
--radio-map-noise-figure-db=7
--disable-radio-map-scene-render
--disable-radio-map-stats-plot
```

The derived link metrics use:

```text
RX power dBm = TX power dBm + RX path gain dB
Path loss dB = -path gain dB
Noise floor dBm = -174 + 10*log10(bandwidth Hz) + noise figure dB
SNR dB = RX power dBm - noise floor dBm
Geometric RTT = 2 * 3D_TX_RX_distance / speed_of_light
Ray RTT = 2 * minimum cached Sionna ray delay, when available
```

## Notes

The LTE version uses `car_1` for the UE/source node and `car_2` for the eNB node, matching the repository convention where ns-3 node IDs map through `obj<N+1>` to Sionna scene objects named `car_N+1`.

The TAP is attached to a separate CSMA device on the ghost node, not directly to the LTE device.
